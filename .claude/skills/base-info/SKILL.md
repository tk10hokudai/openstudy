## 2. 出題カテゴリと比率

### 科目A（section_id=1）
全 60 問、120 分。分野別の出題比率は以下のとおり。

| 大分類 | 小分類（例） | 比率 |
|---|---|---|
| テクノロジ系 | 基礎理論、アルゴリズム、データ構造 | 約 50% |
| テクノロジ系 | コンピュータ構成、ソフトウェア、ネットワーク、セキュリティ、DB | |
| マネジメント系 | プロジェクト管理、サービス管理 | 約 20% |
| ストラテジ系 | 経営戦略、システム戦略、法務 | 約 30% |

問題を追加するときは、上記カテゴリのバランスを意識する。
特にテクノロジ系が薄い場合は補強する。

### 科目B（section_id=2）
全 20 問、120 分（合計科目Aと共通の 120 分枠）。

| 分野 | 問題数目安 |
|---|---|
| アルゴリズム・プログラミング（擬似言語） | 16 問（80%） |
| 情報セキュリティ | 4 問（20%） |

アルゴリズム問題は以下のテーマをバランスよく出題する:
- 基本制御構造（for/while/if）
- 配列操作（探索・ソート・最大最小）
- 関数・再帰
- スタック・キュー操作
- 文字列処理
- ビット演算
- 二次元配列・行列

## 3. IPA 疑似言語フォーマット（科目B必須）

### 3-1. 基本構文ルール

```
// 変数宣言（型: 変数名 ← 初期値）
整数型: x ← 0
整数型: i, j          // 複数変数はカンマ区切り
実数型: ave
文字列型: str ← "HELLO"
論理型: flag ← false
整数型: arr[5]        // 配列宣言
整数型: arr[] ← {3, 1, 4, 1, 5}  // 初期値付き配列
```

### 3-2. 演算子

| 種別 | 記号 |
|---|---|
| 代入 | `←` |
| 加減乗除 | `+`, `-`, `×`, `÷` |
| 剰余 | `mod` |
| 比較 | `=`, `≠`, `<`, `>`, `≦`, `≧` |
| 論理 | `and`, `or`, `not` |
| べき乗 | `^` または `**` （問題文で定義） |

**注意**: `≤` `≥` は使わず、必ず `≦` `≧` を使う。`AND`/`OR` ではなく小文字 `and`/`or`。

### 3-3. if 文

```
if (条件式)
    処理
elseif (条件式)
    処理
else
    処理
endif
```

**括弧は必須**。`elseif` は一語。

### 3-4. while 文

```
while (条件式)
    処理
endwhile
```

### 3-5. for 文

```
for (変数 を 開始値 から 終了値 まで 増分 ずつ増やす)
    処理
endfor

// 減算の場合
for (変数 を 開始値 から 終了値 まで 減分 ずつ減らす)
    処理
endfor
```

**`for i ← 1 to n` は使わない**。必ず日本語形式。

### 3-6. 関数定義

```
// 戻り値ありの関数
○整数型: 関数名(整数型: 引数1, 整数型: 引数2)
    // 変数宣言
    整数型: ローカル変数
    // 処理
    return 戻り値

// 戻り値なしの手続き
○関数名(整数型: 引数)
    // 処理
```

- `○`（U+25CB）記号を先頭に付ける。
- `endfunction` は書かない（インデントで関数本体を示す）。
- 本体は 4 スペースインデント。
- 呼び出し側: `整数型: result ← 関数名(引数)`

### 3-7. 配列アクセス

- インデックスは **1 始まり**（0始まりは使わない）
- `arr[i]`、二次元: `matrix[i][j]`
- 末尾への追加: `arr の末尾に value を追加する`

### 3-8. インデント規則

- **4 スペース**をインデント単位とする
- ネストするたびに 4 スペース追加
- `endif`/`endwhile`/`endfor` はブロック開始キーワードと同じ深さ

### 3-9. コメント

```
// これはコメント
```

## 4. 科目B問題の作成手順

1. **テーマ選択**: 現在の問題一覧を確認し、不足しているテーマを選ぶ。
2. **アルゴリズム設計**: 問題として成立するアルゴリズムを考える。
   - コードは 15〜30 行程度が適切
   - トレース可能な明確な計算過程があること
3. **疑似言語コード作成**: 上記フォーマットに厳密に従う。
4. **body_text の構造（重要）**: UI が問題文とコードを別々に表示するため、以下の形式を守る。
   - **1行目〜**: 問題文（「次の擬似コードを実行したとき、…」）
   - **空行1行**: 問題文とコードの区切り（`\n\n` で分割される）
   - **それ以降**: 疑似言語コード本体
   - SQL 文字列リテラル内では実際に改行を入れて記述する（`\n` エスケープは使わない）
5. **選択肢作成**:
   - 正解を 1 番目に記述
   - 誤答は「ありがちな計算ミス・オフバイワン・ループ数の勘違い」から選ぶ
   - 4 択（正解 1 + 誤答 3）
6. **解説作成**: 変数の値の遷移をステップごとに記述する。
7. **自己検査**:
   - [ ] 疑似言語フォーマットが正しいか（括弧、`≦`/`≧`、for 文）
   - [ ] body_text の構造が正しいか（問題文 + 空行 + コード）
   - [ ] コードを手動トレースして答えが合うか
   - [ ] 誤答が「正解ではない」ことを確認
   - [ ] 解説が正解の根拠を示しているか
   - [ ] `question_number` が既存の連番に続いているか
   - [ ] 選択肢の正解が 1 番目か

## 5. 科目A問題の作成手順

1. **カテゴリ選択**: 現在の問題分布を確認し、薄いカテゴリから追加する。
2. **問題文作成**: 定義・特徴・違いを問う形式が基本。
3. **選択肢作成**:
   - 正解を 1 番目に記述
   - 誤答は「試験でよく混同される概念」から選ぶ
4. **解説**: 正解の根拠と、主な誤答が誤りである理由を含める。

## 7. よくある間違いチェックリスト

| 間違い | 正しい形 |
|---|---|
| `for i ← 1 to n` | `for (i を 1 から n まで 1 ずつ増やす)` |
| `while n < 100` | `while (n < 100)` （括弧必須） |
| `if x = 1` | `if (x = 1)` （括弧必須） |
| `function f(n)` / `endfunction` | `○整数型: f(整数型: n)` / endfunction なし |
| `AND` / `OR` | `and` / `or` （小文字） |
| `≤` / `≥` | `≦` / `≧` |
| 0始まりインデックス `arr[0]` | `arr[1]` から始める |
| 正解を 2〜4 番目に配置 | 正解は必ず **1 番目** |

### 2026-04-15: 科目B コードの UI 表示崩れ修正（body_text 構造ルール追加）
**指摘内容**: 科目Bのコードが改行・インデントなしで横一列に表示され、読めない状態だった。
**根本原因**:
1. `body_text` の改行コードが DB に CRLF (`\r\n`) で保存されており、UI の `indexOf('\n\n')` が `\r\n\r\n` を検出できなかった。
2. 結果としてコードが `<pre>` ではなく `<div>` に入り、等幅フォント・改行保持が効かなかった。
**修正内容**:
- `app/exams/[examId]/quiz/page.tsx` の `renderBodyText` 関数に CRLF→LF 正規化を追加。
  ```ts
  const normalized = bodyText.replace(/\r\n/g, '\n').replace(/\r/g, '\n');
  ```
- UI は `\n\n`（空行1行）を境界として問題文を `<div class="question-text">`、コードを `<pre class="code-block">` に分けて描画する。
**作問への適用範囲**:
- 科目B の `body_text` は「問題文 + 空行1行 + 疑似言語コード」の形式を必ず守ること（セクション 4 および SQL テンプレートに反映済み）。
- 空行が抜けるとコードが等幅フォントで表示されないため、SQL リテラル内の改行を必ず確認する。
